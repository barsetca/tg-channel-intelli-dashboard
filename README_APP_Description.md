# Telegram Channel Intelligence Dashboard — описание работы приложения

## 1. Введение

### Цель системы
Система предназначена для ускорения поиска, сбора, анализа и сравнения Telegram-каналов, а также для семантической работы с накопленным контентом.  
Продукт объединяет в одном интерфейсе источники данных (Telegram live + локальный каталог), AI-планирование, аналитику, аудит и ручную проверку спорных случаев.

### Для кого (ЦА)
- маркетинговые команды и агентства;
- продуктовые и контент-аналитики;
- исследовательские и медиа-команды;
- специалисты, которые регулярно делают конкурентный анализ Telegram-каналов.

### Какую проблему решает
- ручной поиск и оценка каналов занимают много времени;
- данные по каналам и постам разрознены;
- сложно анализировать собственные каналы и сравнивать из с другими;
- сложно повторяемо фиксировать, почему система приняла то или иное решение;
- без журнала аудита трудно проверять ошибки и качество.

### Ценность
- быстрый переход от запроса к списку релевантных каналов;
- автоматизация рутинных шагов (сбор, нормализация, первичная аналитика, сравнение и анализ);
- прозрачность через `audit_runs` и `audit_run_items` за счет подтверждаемого следа выполнения;
;
- безопасная деградация через `needs_review` вместо "тихих" ошибок, что защищает пользователя от ложной уверенности в результате;

### Мини-экономика проекта (оценка)
Пример для 100 операций (поиск/проверка/сводка/сравнение), при стоимости часа аналитика 1500 руб:

- **До внедрения**
  - 1 ручная операция: ~18 мин
  - 100 операций: 1800 мин = 30 ч
  - стоимость: ~45 000 руб
- **После внедрения**
  - 1 операция в системе: ~5 мин
  - 100 операций: 500 мин = 8.3 ч
  - стоимость: ~12 450 руб
- **Эффект**
  - экономия времени: ~21.7 ч на 100 операций
  - экономия бюджета: ~32 550 руб на 100 операций
  - снижение риска ошибок и потерь контекста за счет аудита и ручной проверки.

Вывод: ценность системы в том, что она сокращает время на рутинную аналитику, делает процесс воспроизводимым и снижает операционные риски.

## 2. Подробное описание функциональных сценариев

Ниже описаны сценарии из главного меню и вложенные пользовательские потоки.

### Как читать алгоритмы ниже
- В каждом сценарии показана связка: **что делает пользователь в UI → какой endpoint вызывается → что делает backend → где подключается LLM/RAG → что возвращается в интерфейс**.
- Где уместно, приведены **типовые формы запросов к LLM** (семантически и структурно). Формулировки могут слегка отличаться от рантайма, но логика и ожидаемый формат ответа совпадают.
- `needs_review=true` — это управляемый стоп-сигнал: система не "притворяется", что всё выполнено, а просит уточнить запрос и записывает причину в аудит.

## Сценарий 1. Поиск каналов

### Цель
Найти релевантные каналы по теме и фильтрам, либо в сохраненном каталоге, либо в Telegram live.

### 1) Действия пользователя
- Пользователь открывает раздел `Поиск каналов`.
- Выбирает источник:
  - `Сохраненный каталог`;
  - `Telegram live`.
- Заполняет фильтры (тема/ниша, количество, язык, страна, дополнительные условия, и т.д.).
- Для сохраненного каталога может выбрать `Тема / ниша` из выпадающего списка (включая `Все`).
- Нажимает `Найти каналы`.

### 2) Алгоритм работы программы
1. Пользователь отправляет форму, frontend вызывает `POST /api/v1/search-channels`.
2. Backend делает первичную проверку:
   - корректность полей;
   - конфликтность/полнота фильтров;
3. Если есть несоответствия, включается quality-gate:
   - формируется `manual_review` с причиной и подсказками;
   - в UI пользователь видит понятный текст, что нужно уточнить.
4. Если можно продолжать, вызывается AI-планировщик поиска.
   - Типовой смысл запроса к LLM: "Преобразуй пользовательский запрос в параметры поиска каналов, верни структурированный JSON".
   - Типовой ожидаемый ответ:
```json
{
  "search_topic": "инвестиции в недвижимость",
  "min_subscribers": 10000,
  "max_subscribers": 300000,
  "language": "ru",
  "region_country": "KZ",
  "count": 20,
  "channel_type": "all",
  "confidence": "high"
}
```
- Если пользователь заполнил поле Дополнительные условия, то делается дополнительный вызов LLM _review_extra_conditions(body) (LLM-проверка конфликтов темы и доп. условий) 
5. Далее ветвление по источнику:
   - `saved_catalog`: быстрый SQL-поиск по локальному каталогу;
   - `telegram_live`: запуск оркестратора и staged-пайплайна:
     - **planner stage**: уточнение параметров запуска;
     - **telethon ingest**: сбор кандидатов из Telegram;
     - **sqlite persist**: запись каналов и records;
     - **metrics**: расчет частоты/активности;
     - **ai pipeline**: уточнение тематики каналов;
     - **vector stage**: подготовка данных для семантических сценариев.
6. Что видит пользователь в UI:
   - либо мгновенный список (saved catalog),
   - либо модалку фоновой задачи с этапом и статусом (`queued/running/completed/failed`).
7. Что сохраняется:
   - `audit_runs` — общий "паспорт" запуска;
   - `audit_run_items.snapshot_json` — records (снимки строк результата).

### 3) Результат для пользователя
- Возвращается список каналов и/или background job.
- В UI отображаются карточки/таблица, либо статус фоновой задачи.
- Доступны следующие шаги: выбрать канал, перейти к анализу, сохранить/использовать в наборах данных.

### Негативные кейсы
- `needs_review=true`: показывается причина и подсказки, автоматический сбор не запускается.
- Недоступен Telegram live/сессия: возвращается failed-статус фоновой задачи.
- Некорректные фильтры: ошибка валидации (`422`).

### Резюме
Сценарий закрывает "быстрый вход" в аналитику: от запроса к подборке каналов с аудируемым результатом.

## Сценарий 2. Наборы данных (dataset ≈ channel)

### Цель
Управлять рабочим набором каналов: создать, выбрать, собрать обновления, удалить.

### 1) Действия пользователя
- Пользователь открывает `Наборы данных`.
- Видит таблицу сохраненных каналов (как datasets).
- Доступные действия:
  - `Создать канал`;
  - `Собрать` (для выбранного канала);
  - `Удалить`;
  - `Очистить выбор`.
- При создании вводит username/ссылку, тему, доп. условия.

### 2) Алгоритм работы программы
1. Таблица datasets загружается из `GET /api/v1/channels?limit=&offset=`.
2. Создание dataset/channel (`POST /api/v1/channels`):
   - нормализация `channel_ref` (`@name` / `t.me/name`);
   - проверка на дубликат по `telegram_id` и `username`;
   - если канал уже есть — возвращается `already_exists=true`, и UI показывает отдельную модалку с понятным объяснением.
3. Сбор по выбранному каналу (`POST /api/v1/channels/{channel_id}/collect`):
   - берутся параметры формы (`channel_ref`, `topic`, `extra_conditions`);
   - внутри формируется вызов live-поиска в режиме saved channel;
   - запускается тот же оркестрационный пайплайн, что и в live-поиске.
4. Точки AI/LLM:
   - LLM используется в planner-части и в AI-stage пайплайна для нормализации/уточнения темы и релевантности;
   - результат LLM всегда приводится к структурированным полям, чтобы UI не зависел от "свободного текста".
5. Ошибки и аудит:
   - `404` — канал не найден;
   - `502` — ошибка внешнего сервиса;
   - `500` — внутренняя ошибка;
   - при `502/500` обязательно создается `audit_runs` (`action=dataset_collect_error`) с текстом и кодом ошибки.
6. Пользовательский эффект:
   - UI не блокируется;
   - пользователь видит статус задачи и может продолжать работу;
   - records становятся видны через "Витрину данных".

### 3) Результат для пользователя
- Модальные сценарии:
  - успешное создание;
  - канал уже существует;
  - нуждается в ручной проверке;
  - фоновая задача сбора.
- Пользователь видит понятный статус и может сразу продолжить действия.

### Негативные кейсы
- Повторное создание существующего username: запись не дублируется, показывается отдельное сообщение.
- Ошибка создания/сбора: показывается модалка ошибки.
- `needs_review` при collect: сбор не идет автоматически, предлагается переход в `Требует проверки`.

### Резюме
Сценарий делает "управляемую операционную полку" для каналов и связывает ручную работу пользователя с формальным аудитом.

## Сценарий 3. Анализ канала

### Цель
Получить аналитический профиль конкретного канала.

### 1) Действия пользователя
- Пользователь открывает `Анализ канала`.
- Вводит username/ссылку или выбирает канал из сохраненных.
- Вводит количество постов (до 20) для анализа
- Запускает анализ.

### 2) Алгоритм работы программы
1. Запрос из UI:
   - `POST /api/v1/analyze/by-handle` (если ввели username/ссылку),
   - или `POST /api/v1/analyze/{channel_id}` (если выбрали сохраненный канал).
2. Проверка доступа и данных:
   - существует ли канал;
   - есть ли минимум данных для аналитики;
   - если нет — `needs_review` с объяснением.
3. Сбор контекста:
   - описание канала;
   - последние посты;
   - метрики активности/вовлеченности.
4. Точка LLM (основной аналитический промпт):
   - смысл запроса: "Проанализируй канал и верни структурированный отчет: сильные стороны, риски, контент-стратегия, tone, рекомендации".
   - ожидается структурированный JSON (или строго шаблонизированный объект), который затем рендерится на UI блоками.
   - типовая форма ответа:
```json
{
  "summary": "Канал про ...",
  "strengths": ["..."],
  "risks": ["..."],
  "content_strategy": ["..."],
  "tone_of_voice": ["..."],
  "recommendations": ["..."]
}
```
5. Сохранение:
   - результат записывается в `analyses`;
   - ключевые артефакты связываются с каналом для повторного открытия.
6. Что видит пользователь:
   - на странице аналитики сразу появляются структурированные секции отчета;
   - пользователь может перейти к анализу постов
   - на странице доступен список ранее созданных отчетов по всем каналам, с возможностью их посмотреть

### 3) Результат для пользователя
- Отчет на экране с метриками и текстовыми выводами.
- Возможность повторного анализа
- Возможность открыть предыдущие отчеты изи списка

### Негативные кейсы
- Канал недоступен/не найден: `needs_review` с причиной.
- Недостаточно данных для анализа: частичный отчет или ошибка с подсказкой.

### Резюме
Сценарий переводит "сырой канал" в структурированный аналитический отчет.

## Сценарий 4. Сводка последних постов (резюме постов)

### Цель
Быстро получить краткое содержание последних публикаций канала.

### 1) Действия пользователя
- В рамках анализа пользователь задает количество постов.
- Нажимает Резюмировать посты  для формирование сводки.

### 2) Алгоритм работы программы
1. Из UI уходит запрос на сводку по выбранному каналу и количеству постов.
2. Backend получает посты из Telegram/БД и подготавливает текст:
   - удаляет технический шум;
   - при длинных текстах делает chunking.
3. Точка LLM (summarization):
   - смысл запроса: "Сделай краткую, фактическую сводку без выдумывания фактов, выдели повторяющиеся темы и сигналы";
   - ожидается компактный структурированный результат (summary + key points).
4. Сводка сохраняется в аналитических сущностях и может использоваться в дальнейшем поиске/сравнении.
5. На UI пользователь получает "читаемый за минуту" блок вместо десятков исходных постов.

### 3) Результат для пользователя
- Краткая сводка: ключевые темы, идеи, повторяющиеся паттерны.
- Возможность семантическогопоиска (см. ниже)

### Негативные кейсы
- Нет постов за период: информирование пользователя.
- Ошибка внешнего LLM/API: возврат ошибки и запись в журнал выполнения.

### Резюме
Сценарий экономит время на первичное понимание содержания канала.

## Сценарий 5. Семантический поиск

### Цель
Находить релевантные каналы/фрагменты по смыслу, а не только по ключевым словам.

### 1) Действия пользователя
- Пользователь открывает `Семантический поиск`.
- Вводит вопрос на естественном языке.
- Запускает поиск.

### 2) Алгоритм работы программы
1. Пользователь задает вопрос естественным языком, UI вызывает `POST /api/v1/semantic-search`.
2. Quality-gate проверяет вопрос:
   - если он слишком общий/противоречивый, возвращается `needs_review`.
3. Если запрос корректный, включается RAG-процесс:
   - строится embedding пользовательского вопроса;
   - выполняется similarity-поиск по векторной базе (Qdrant);
   - выбираются top-k релевантных фрагментов.
4. Архитектура баз данных:
   - классический SQL-поиск работает по точным совпадениям;
   - векторный поиск находит смысловые соответствия (синонимы, переформулировки, контекст).
5. Точка LLM synthesis:
   - в LLM передается вопрос + извлеченные источники (retrieved context);
   - смысл запроса: "Сформируй ответ только на основе переданных источников, добавь ссылки/доказательства, не выдумывай факты".
   - результат: человекочитаемый ответ + список источников.
6. Что видит пользователь:
   - итоговый ответ с объяснением "почему именно эти каналы/посты";
   - если `needs_review`, то понятную причину и шаги по уточнению запроса.

### 3) Результат для пользователя
- Ответ по существу + источники.
- При `needs_review` — причина и дальнейшие шаги.

### Негативные кейсы
- Пустой/двусмысленный вопрос: `needs_review=true`.
- Нет релевантных данных: корректный пустой ответ с пояснением.

### Резюме
Семантический поиск дает "интеллектуальный слой" поверх накопленной базы.

## Сценарий 6. Сравнение каналов

### Цель
Сравнить 2 - 3 канала из сохраненных по метрикам и рекомендациям.

### 1) Действия пользователя
- Пользователь открывает `Сравнение`.
- Выбирает каналы для сравнения.
- Запускает сравнение.

### 2) Алгоритм работы программы
1. Из UI отправляется `POST /api/v1/channels/compare` с выбранными каналами.
2. Backend собирает сопоставимые метрики из БД и анализов.
3. Точка LLM-интерпретации:
   - смысл запроса: "Сравни каналы по одинаковым метрикам, выдели сильные/слабые стороны и практические рекомендации";
   - LLM возвращает структурированный вывод, который UI показывает в блоках.
4. Пользователь получает не только цифры, но и интерпретацию "что это значит для решения".

### 3) Результат для пользователя
- Сравнительная таблица/карточка.
- Выводы: сильные стороны и риски по каждому каналу.

### Негативные кейсы
- Недостаточно выбранных каналов.
- Нет данных по одному из каналов.

### Резюме
Сценарий помогает принять решение "какой канал лучше для цели" или сравнить свой канал с конкурентами

## Сценарий 7. Витрина данных

### Цель
Просматривать нормализованные records и выгружать их.

### 1) Действия пользователя
- Пользователь открывает `Витрина данных`.
- Выбирает лимит, просматривает записи.
- Экспортирует в JSON/CSV.

### 2) Алгоритм работы программы
1. `GET /api/v1/data-showcase?limit=...`
2. `GET /api/v1/data-showcase/export?format=json|csv&limit=...`
3. Backend читает строки из `audit_run_items` + `audit_runs`, формирует витрину.

### 3) Результат для пользователя
- Доступен список records с источником и временем.
- Экспортируемые файлы для отчетности.

### Негативные кейсы
- Пустая витрина: корректное сообщение "данных нет".
- Ошибка экспорта: сообщение с кодом и текстом ошибки.

### Резюме
Сценарий дает "операционную прозрачность" и формат выгрузки результата.

## Сценарий 8. Требует проверки (Manual Review)

### Цель
Централизованный журнал кейсов, где автоматический запуск небезопасен.

### 1) Действия пользователя
- Пользователь открывает `Требует проверки`.
- Фильтрует по источнику (`all`, `audit`, `search`, `analyze`, `semantic`).
- Просматривает причины и подсказки.
- Экспортирует в JSON/CSV.

### 2) Алгоритм работы программы
1. `GET /api/v1/manual-review?source=...&limit=...`
2. `GET /api/v1/manual-review/export?format=json|csv&source=...&limit=...`
3. Backend объединяет `needs_review` события из audit/search/analyze/semantic.

### 3) Результат для пользователя
- Журнал ручной проверки с причиной, подсказками, временем.
- Понятный список задач, которые нужно уточнить вручную.

### Негативные кейсы
- Неверный фильтр: fallback на `all`.
- Пустой журнал: информативный пустой экран.

### Резюме
Сценарий снижает риск неверных автоматических действий и повышает управляемость качества.

## Вложенные сервисные сценарии

### Сценарий A. AI planning adapter
- Endpoint: `POST /api/v1/ai/plan_and_collect`
- Вход: `{ "query": "string" }`
- Выход: строгий JSON:
  - `plan_steps`
  - `api_url`
  - `fields_to_keep`
  - `confidence`
  - `needs_review`
- Назначение: формализовать план запроса до запуска фактического сбора.

Типовой смысл запроса к LLM в адаптере:
- "Преобразуй пользовательский query в 1-5 конкретных шагов плана и параметры вызова API. Верни строго JSON без пояснений".

Типовой ответ:
```json
{
  "plan_steps": ["Уточнить тему", "Определить фильтры", "Запустить поиск в каталоге", "При нехватке данных перейти в live"],
  "api_url": "/api/v1/search-channels",
  "fields_to_keep": ["id", "username", "title", "subscriber_count", "primary_topic", "last_post_at"],
  "confidence": "medium",
  "needs_review": false
}
```

Что это дает пользователю на UI:
- можно объяснить "почему система собирается делать именно это";
- при `confidence=low` пользователь получает честный сигнал уточнить запрос до старта сбора.

### Сценарий B. Состояние API и интеграции
- `GET /api/v1/health`
- `GET /api/v1/telegram/status`
- Используется для оперативной диагностики доступности платформы.

### Сценарий С. Список дополнительных возможностей
1. Поиск каналов в режиме Telegram (live) позволяет создавать или обновлять до 15 новых каналов за один запрос
2. Список каналов полученный через меню Поиск каналов в режиме Сохраненый катлог является интерактивным и содержит:
- быстро удалить канал через икинку корзины
- перейти в карточку канала, кликнув на строке канала
3. Карточка канала содержит информацию о канале и предоставляет доступ к сервисам для этого канала:
- Открыть канал в телеграм
- Провести аналитику канала
- Анализировать канал 
- Найти похожие каналы среди сохраненных
- Сделать краткое резюме от 5 до 20 последних постов канала

## 3. Заключение

Система реализует полный цикл: от постановки задачи пользователем до получения структурированного, аудируемого результата.  
Архитектура сочетает:
- транзакционную часть (SQLite + ORM),
- оркестрацию фоновых задач,
- AI-компоненты (планирование, анализ, summary),
- RAG/семантический слой,
- прозрачный аудит и ручную проверку.

Итог: приложение закрывает как "позитивные" рабочие сценарии (поиск, анализ, сравнение, экспорт), так и "негативные" (ошибки, двусмысленные запросы, внешние сбои) с контролируемым поведением и понятной обратной связью для пользователя.

Отдельно важно: LLM в системе не "магический черный ящик", а управляемый этап в четком пайплайне.  
Пользователь на UI видит результат в понятных сущностях (список каналов, отчет, сводка, сравнение, журнал проверки), а разработчик и оператор всегда могут восстановить ход выполнения через аудит.
